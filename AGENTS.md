# Landing Maker Agents

本项目使用四个主 Agent 协作创建 Landing 页面，并整合两个设计相关 Skill：

| Agent | Skill | 职责 |
|-------|-------|------|
| 入口 Agent（产品专家） | `landing-orchestrator` | 通过产品文档解析和多轮对话深度挖掘需求，生成 demand.md；确认设计路径；**不负责协调管理和项目复盘**（由项目经理负责） |
| **项目经理** | `landing-project-manager` | **专门负责协调管理**：组织头脑风暴、任务编排、流程管理、迭代协调、质量把关、项目复盘；不负责需求挖掘和具体的设计、开发、SEO 工作 |
| 设计大师 | `landing-design-master` | 路径 1、3 主导设计，产出设计规格；**路径 2 下积极参与，提供具体设计建议** |
| SEO 专家 | `landing-seo-expert` | 基于 demand.md 生成关键词与 SEO 建议，供前端专家进行搜索引擎优化 |
| 前端专家 | `landing-frontend-expert` | 路径 2 主导实现；路径 1、3 实现设计规格；**路径 2 下主动参考设计大师建议，提升设计质量** |
| 评分 Agent | `landing-scorer` | 按固定标准对页面打分，输出结构化评分报告 |
| **数据收集专家** | `landing-data-collector` | **后台知识维护**：采集和精炼设计语言、实现模式、行业解决方案，持续丰富产品、设计和前端三个知识库；与主流程解耦，可定时触发或按需触发 |

**整合的设计 Skill**：
- `frontend-design`：前端美学原则，避免泛化 AI 风格
- `ui-ux-pro-max`：UI/UX 规范、可访问性、Pre-Delivery Checklist

## 知识库体系

Landing Maker 为产品专家、设计大师和前端专家分别建立了 DAG（有向无环图）知识库，用于结构化存储和检索专业知识。

### 三个知识库

| 知识库 | 使用方 | 内容方向 | 侧重点 |
|--------|--------|----------|--------|
| **产品知识库** | 产品专家 Agent | 行业解决方案、标准文字内容、需求模板、产品定位模式 | **侧重积累行业的解决方案和标准的文字内容** |
| **设计知识库** | 设计大师 Agent | 设计语言、版式规律、配色体系、行业案例、风格参考 | **侧重积累很多套不同的设计模板** |
| **前端知识库** | 前端专家 Agent | 前端页面范式、展示模式、实现模式、组件库、布局解法、动效实现、兼容性方案 | **侧重积累前端页面和展示模式的标准范式** |

### 知识库位置

- **产品知识库**：`.cursor/knowledge/product-expert/`
- **设计知识库**：`.cursor/knowledge/design-master/`
- **前端知识库**：`.cursor/knowledge/frontend-expert/`

### 知识库使用

**产品专家**：
- 需求挖掘时检索行业解决方案和需求模板
- 生成 `demand.md` 时使用标准文字内容模板
- 头脑风暴时提供行业最佳实践参考

**设计大师**：
- 设计时检索设计语言、版式规律、配色体系
- 路径 3（参考+创意）时快速找到相似风格的设计模板
- 头脑风暴时提供设计角度的建议和参考

**前端专家**：
- 实现时检索实现模式、组件库、布局解法
- 路径 2（内容工具）时选择合适的展示模式组织内容
- 头脑风暴时提供技术可行性建议

### 知识库维护

**数据收集专家**负责：
- 采集设计语言、产品知识、前端知识
- 精炼处理，生成 DAG 节点
- 更新知识库（knowledge-dag.json + 知识文件）
- 更新 CHANGELOG.md
- 记录 COLLECTION_LOG.md

**知识库更新流程**：
1. 数据收集专家采集新知识
2. 精炼处理，生成 DAG 节点
3. 更新知识库（knowledge-dag.json + 知识文件）
4. 更新 CHANGELOG.md
5. 记录 COLLECTION_LOG.md
6. 后续项目自动使用更新后的知识库

### DAG 结构

- **节点**：知识点/模式（如「极简风格」「Hero 区块」「响应式布局」）
- **边**：依赖关系（如「极简风格」依赖「留白」「无衬线字体」）
- **检索**：支持按语义和依赖链检索

**知识库关系**：
- 产品知识库 → 设计知识库：产品定位影响设计风格选择
- 设计知识库 → 前端知识库：设计规格影响实现方式
- 三个知识库相互关联，形成完整的知识体系

## 三种设计路径

| 路径 | 名称 | 主导方 | 适合场景 |
|------|------|--------|----------|
| **路径 1** | 纯创意 | 设计大师 | 追求特别、有设计感，创意优先 |
| **路径 2** | 内容工具 | 前端专家 | 内容已准备好，需要合理结构充分展示 |
| **路径 3** | 参考+创意 | 设计大师 | 有参考页面，希望在此基础上创新 |

## 使用方式

在 Cursor 中描述需求，**产品专家 Agent** 会通过多轮对话逐步明确需求，例如：

- "帮我做一个 SaaS 产品的 landing 页面" → 产品专家开始多轮对话，挖掘需求
- 用户可提交产品文档（PRD、品牌手册等）→ 产品专家解析文档并与对话信息融合
- "我要路径 1，做一个极简风格的产品页" → 产品专家继续深入挖掘路径 1 相关需求
- "路径 2，内容已经整理好了，帮我排成页面" → 产品专家确认内容细节和结构
- "路径 3，参考这个网站 [URL]，做一个类似风格的新品介绍页" → 产品专家分析参考并明确创新点
- "给 tests/20250209-160500/landing.html 打个分" → 评分 Agent
- "根据这个链接 [URL] 提炼关键信息并更新知识库" → 数据收集专家执行完整采集流程，更新知识库

输出到 `tests/YYYYMMDD-HHMMSS/` 目录：新上下文创建新文件夹，每个子文件夹含 `demand.md`（用户需求）、`seo-suggestions.md`（SEO 建议）和生成的页面。**产品专家通过多轮对话和文档解析生成 `demand.md`**，生成前会向用户展示并确认。用户同意后，**项目经理组织头脑风暴**，各专家提出建议，产品专家更新需求。然后**项目经理协调 SEO 专家基于 `demand.md` 生成 `seo-suggestions.md`**，进入生成阶段。**每次生成完成后**，项目经理调用评分 Agent 对页面评分，并将报告写入同目录下的 `rating.md`。每次调整生成新文件（`landing.html`、`landing-v2.html` 等）时，会重新评分并更新 `rating.md`。**项目完成后**，项目经理组织项目复盘，生成 `retrospective.md`（项目复盘报告）。
